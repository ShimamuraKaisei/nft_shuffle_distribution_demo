# 要件定義書: GitHub → Notion ドキュメント同期システム

## 1. 背景と目的

SUSHITOP では複数のリポジトリに技術ドキュメント（Markdown）が存在するが、非エンジニアメンバーが GitHub 上のドキュメントを直接参照するのは現実的でない。

本システムは、任意のリポジトリに導入可能な GitHub → Notion 一方向同期の仕組みを提供し、**開発者は Git ワークフローの中でドキュメントを管理し、非エンジニアは Notion 上で閲覧する**という運用を実現する。

## 2. スコープ

### 対象

- GitHub リポジトリ上の指定された Markdown ファイル（1リポジトリにつき1ファイル）を、Notion 上の指定ページに自動同期する

### 対象外

- Notion → GitHub の逆方向同期
- Notion 上でのドキュメント編集（閲覧専用）
- 画像・PDF 等バイナリファイルの同期
- 複数ファイルの一括同期・Notion データベースへのマッピング

## 3. 機能要件

### 3.1 同期トリガー

| 項目 | 仕様 |
|---|---|
| トリガー | 対象ブランチへの PRマージ |
| 対象ブランチ | `main`（リポジトリごとに変更可能） |
| 起動条件 | 指定パスの Markdown ファイルに変更があった場合のみ |
| 手動実行 | GitHub Actions の `workflow_dispatch` による手動トリガーも可能とする |

### 3.2 同期処理

| 項目 | 仕様 |
|---|---|
| 同期方式 | 全置換（既存ブロック全削除 → 新規ブロック追加） |
| 変換ライブラリ | `@tryfabric/martian`（Markdown → Notion ブロック変換） |
| ブロック上限 | Notion API の1リクエスト100ブロック制限に対応し、チャンク分割で送信する |
| 冪等性 | 同一内容で複数回実行しても結果が変わらないこと |

### 3.3 対応する Markdown 構文

| 構文 | 対応 |
|---|---|
| 見出し（h1〜h3） | ○ |
| 段落・改行 | ○ |
| 箇条書きリスト | ○ |
| 番号付きリスト | ○ |
| 太字・斜体・取り消し線 | ○ |
| インラインコード | ○ |
| コードブロック（言語指定含む） | ○ |
| 引用 | ○ |
| リンク | ○ |
| テーブル | ○ |
| 画像（外部 URL のみ） | ○ |
| 画像（ローカルパス参照） | × 対象外 |
| HTML 埋め込み | × 変換されない |
| Mermaid 等の拡張構文 | × 変換されない |

### 3.4 エラーハンドリング

| 項目 | 仕様 |
|---|---|
| リトライ | Notion API の 429（Rate Limit）レスポンスに対し、`Retry-After` ヘッダに従い最大3回リトライする |
| タイムアウト | GitHub Actions ジョブのタイムアウトを5分とする |
| 失敗時の挙動 | GitHub Actions のステータスを `failure` にし、既存の Notion ページは変更しない（削除途中で失敗した場合を除く） |
| ログ | 処理対象ファイル名、生成ブロック数、削除ブロック数、API エラー詳細を Actions ログに出力する |

## 4. 非機能要件

| 項目 | 仕様 |
|---|---|
| 導入容易性 | 任意のリポジトリに3ファイル（workflow yml, スクリプト, package.json）を追加し、2つの Secret を設定するだけで動作すること |
| 実行環境 | GitHub Actions（ubuntu-latest）、Node.js 20 |
| 依存パッケージ | `@notionhq/client`, `@tryfabric/martian` の2つのみ |
| 実行時間 | 通常のドキュメント（200ブロック以下）で1分以内に完了すること |
| セキュリティ | Notion API Key は GitHub Secrets 経由でのみ参照し、ログに出力しない |

## 5. 設定項目

各リポジトリで変更が必要な設定値は以下の通り。

### GitHub Secrets（リポジトリごとに設定）

| Secret 名 | 説明 |
|---|---|
| `NOTION_API_KEY` | Notion Internal Integration Token |
| `NOTION_PAGE_ID` | 同期先の Notion ページ ID |

### ワークフロー内の定数（yml 内で変更）

| 変数 | 説明 | デフォルト値 |
|---|---|---|
| `MARKDOWN_PATH` | 同期対象の Markdown ファイルパス | `docs/your-document.md` |
| 対象ブランチ | PRマージを監視するブランチ | `main` |
| `paths` フィルター | Actions 起動条件のファイルパス | `MARKDOWN_PATH` と同一 |

## 6. 制約事項

- **全置換方式のため、Notion 側のコメント・リアクション・手動編集内容は PRマージのたびに消失する。** 非エンジニアには「このページは自動同期されるため編集しないこと」を周知する必要がある。
- Notion API のブロック削除は1件ずつ逐次実行のため、ブロック数が多い場合は Rate Limit に達する可能性がある。リトライで吸収するが、極端に大きなドキュメント（500ブロック超）では注意が必要。
- `@tryfabric/martian` の変換精度に依存する。未対応の Markdown 拡張構文は無視されるか、プレーンテキストとして出力される。

## 7. 将来の拡張候補

現時点ではスコープ外だが、将来的に需要が発生した場合に検討する項目を記載する。

| 拡張項目 | 概要 | 導入判断の基準 |
|---|---|---|
| 複数ファイル対応 | `docs/**/*.md` を Notion データベースに同期 | 同期対象ドキュメントが3ファイル以上になった時点 |
| 画像の S3 連携 | ローカル画像を S3 にアップロードし URL を差し替え | ドキュメントに図表が必須になった時点 |
| frontmatter 対応 | YAML メタデータから Notion プロパティを自動設定 | データベース方式に移行した時点 |
| 差分同期 | `git diff` で変更ブロックのみ更新 | ドキュメントが大規模化し全置換の実行時間が問題になった時点 |
| Slack 通知 | 同期完了・失敗を Slack チャンネルに通知 | 運用上、同期失敗の検知が遅れる問題が発生した時点 |
